# Deep Semantic Audit — June 2026

Independent fresh-eyes semantic audit of all first-party production source (lib/ Dart + functions/src TS), beyond the structural scanners (which were green). Generated from a multi-agent audit→adversarial-critic→synthesis workflow. Pattern keys are tracked in `patterns.json`; new codified rules land in `../rules.json`.

**Scope:** lib/ = 591 files / ~119k LOC (42 slices); functions/src = 90 files / ~32k LOC (17 slices).

**Status legend:** ✅ fixed in Batch A · ⏪ reverted (false positive) · 📋 backlog (product/infra decision).

## lib/ (client) — 149 verified findings (9 high / 38 med / 51 low)

> The Catch lib/ codebase is, on the whole, mature and disciplined: backend errors route through a normalization pipeline (withBackendErrorContext/normalizeBackendError -> appErrorDescriptor) and typed AppExceptions, Firestore writes generally use partial update()+FieldValue, denormalized counts are backend-owned, services hide behind Riverpod providers, and a custom analyzer plugin already enforces raw-value/token drift. The verified findings cluster into a small number of high-leverage cross-cutting themes rather than scattered one-offs. The single most important theme is error-surface inconsistency: roughly a dozen sites either swallow MutationErrors entirely (swipe action, event_success companion cards, host team/transfer actions), render data-load failures as a dead CatchEmptyState with no retry instead of CatchErrorState/CatchInlineErrorState (calendar, saved_events, dashboard sections, safety blocked-accounts), or show raw/hardcoded SnackBars and bare Text that bypass showCatchErrorSnackBar/ErrorBanner and lose mapped copy (force_update, profile photo, image uploads, host invite links, event_success _ErrorText). Because the canonical primitives already exist and are used correctly elsewhere in the same features, these are codify-and-fix-everywhere opportunities, not redesigns. The highest-severity individual correctness bugs are genuinely serious and localized: the Razorpay 60s timeout can orphan a charged payment with no server reconciliation (no Razorpay webhook exists), cancelled signed-up events leak into the calendar/stats/Next, the explore feed silently drops or swallows personal out-of-city events, the swipe deck stalls with no feedback on a failed write, choosing non-default profile prompts soft-locks onboarding in a redirect loop, the phone number is editable with no OTP re-verification (identity divergence), and the undismissable force-update screen dead-ends if openStore throws. A second structural theme is data-layer duplication and N-plus-1 access: a triplicated Firestore whereIn fan-out/merge plus a thrice-defined _chunks helper, read-modify-set/full-document-set in three event_success write paths, sequential or per-tile-realtime profile reads, and an unkeyed static Mutation shared across directory list tiles. A third theme is dead/orphaned code from incomplete migrations (legacy ScrollableProfile cluster, EventPinRenderer, EligibilityStep, DashboardEmpty, several unwired getters and an entire unwired analytics funnel) and no-op dead ternaries. A fourth, lower-severity theme is grammar corruption from an automated run->event rename that leaked into user-facing compatibility/run-tag copy. The token system (catch_tokens/catch_fonts) is solid but has a font API that silently discards a fontStyle/italic parameter, producing one no-op italic call site. Overall the codebase does not need broad rework; it needs (a) a handful of high-severity correctness fixes, (b) consolidation of error-surface and data-access patterns into shared primitives, and (c) a dead-code sweep. The audit registry itself would benefit from machine-checkable pattern keys and per-pattern lint rules so these recurrences are prevented rather than re-discovered each pass.


### Cross-cutting patterns (26)

- **`error_state_uses_empty_state_no_retry`** (8 sites → rule `ERROR-UI-001`): Data-load failures render CatchEmptyState/hand-rolled cards with no retry instead of CatchErrorState/CatchInlineErrorState
    - *Why:* ERROR-UI-001 requires app-facing load failures to surface mapped copy (appErrorDescriptor) plus a retry affordance. Multiple screens instead route stream/future errors into a dead empty-state or bespoke card, discarding the error object and stranding users on an 'unavailable' panel with no recovery. The canonical CatchErrorState.fromError / CatchInlineErrorState.fromError primitives exist (verified: catch_error_state.dart:25-46 yields descriptor-mapped title/message + retryable-gated retry) and are used correctly in 25+ sibling screens, so this is pure inconsistency.
    - *Fix:* Replace each error-branch CatchEmptyState/bespoke card with CatchErrorState.fromError (full screen), CatchSliverErrorState.fromError (sliver), or CatchInlineErrorState.fromError (compact section), passing the actual error and an onRetry that ref.invalidate()s the failing provider. Reserve CatchEmptyState strictly for genuinely-empty success states.
    - *Seen in:* lib/calendar/presentation/calendar_screen.dart, lib/events/presentation/saved_events_screen.dart, lib/dashboard/presentation/widgets/dashboard_full.dart, lib/dashboard/presentation/widgets/stride_card.dart, lib/safety/presentation/settings_screen.dart
- **`mutation_error_not_surfaced`** (7 sites → rule `ERROR-UI-001`): MutationError is watched only for isPending; failures are silently swallowed with no user feedback
    - *Why:* Several controllers expose a Mutation that the UI watches only via isPending to drive a spinner, never checking hasError. On failure the spinner clears, optimistic state never applies (or a toggle silently reverts), and the user is told nothing — including on the most security-sensitive action (ownership transfer) and the core swipe loop (deck stalls). riverpod 3.3.2 Mutation.run() rethrows, so the fix is to surface the error, not to assume success. The convention exists in-feature (questionnaire section, filters_screen).
    - *Fix:* Either wrap the .run() call in try/catch + showCatchErrorSnackBar(context, error), or add a mutation.hasError branch rendering ErrorBanner.fromError / a ref.listen MutationErrorSnackbarListener at the screen level. Standardize on one of these two per surface.
    - *Seen in:* lib/swipes/presentation/swipe_screen.dart, lib/event_success/presentation/companion_parts/event_success_companion_feedback.dart, lib/event_success/presentation/companion_parts/event_success_companion_wingman.dart, lib/event_success/presentation/companion_parts/event_success_companion_live_cards.dart, lib/event_success/presentation/live_reveal_parts/event_success_live_reveal_attendee.dart, lib/hosts/presentation/widgets/host_team_management_section.dart
- **`raw_or_hardcoded_snackbar_bypasses_mapped_error`** (10 sites → rule `ERROR-UI-001`): Raw/const SnackBar(Text(...)) or bare Text used for errors, bypassing showCatchErrorSnackBar/ErrorBanner and losing mapped copy
    - *Why:* Transient/inline errors are shown via hand-rolled SnackBars or bare Text with hardcoded strings, discarding the real error object (often an AppException whose authored .message is user-meaningful, e.g. 'That image is too large'). This loses styling, retry, and the permission-vs-network distinction. showCatchErrorSnackBar and ErrorBanner.fromError are the canonical mapped-copy primitives and are used correctly elsewhere.
    - *Fix:* Route transient failures through showCatchErrorSnackBar(context, error) and inline/form errors through ErrorBanner.fromError(error, context: ...); delete the shared local _ErrorText helper in event_success and the parallel raw uploadError-to-string path. Pass the captured error, never a hardcoded string.
    - *Seen in:* lib/core/widgets/rich_share_card_sheet.dart, lib/force_update/presentation/update_required_screen.dart, lib/user_profile/presentation/profile_screen.dart, lib/image_uploads/presentation/photo_upload_controller.dart, lib/hosts/presentation/host_event_manage_screen.dart, lib/event_success/presentation/host_parts/event_success_host_setup.dart, lib/event_success/presentation/live_reveal_parts/event_success_live_reveal_host.dart
- **`read_modify_set_full_document_set`** (3 sites → rule `FIRESTORE-PARTIAL-WRITE-001`): Read-modify-set / full-document set() of client snapshot instead of partial update() of changed keys
    - *Why:* Three event_success write paths read a doc then set() the whole client-held snapshot. This (a) re-emits createdAt/updatedAt as client Timestamp.fromDate, downgrading server-anchored timestamps to host-clock time, and (b) for the two independent opt-out toggles creates a genuine last-write-wins race (they use distinct mutations, so busy-gating does not cross-block). It trips Firestore rules diff() and clobbers concurrent writes — the anti-pattern the project explicitly warns against.
    - *Fix:* Use _ref.set({...changedKeysOnly, 'updatedAt': FieldValue.serverTimestamp()}, SetOptions(merge: true)) or .update() writing only the changed/host-editable keys; reserve full set() for the transaction-guarded initial create. Eliminate the pre-read for preference/compatibility.
    - *Seen in:* lib/event_success/data/event_success_repository/plan.dart, lib/event_success/data/event_success_repository/preference.dart, lib/event_success/data/event_success_repository/compatibility.dart
- **`firestore_whereIn_fanout_and_chunks_duplication`** (6 sites → rule `DESIGN-PRIMITIVES-001`): Triplicated whereIn fan-out/merge stream machinery and a thrice-defined _chunks helper
    - *Why:* The 'watch source stream -> collect ids -> split into 10-id chunks -> fan out one whereIn snapshot subscription per chunk -> emit once all chunks resolve -> teardown' pattern is copy-pasted near-verbatim across three methods in two repositories, and the generic _chunks<T> helper is defined three times. Any leak/teardown/generation-guard fix must be applied in three places, and the clubNameLookupProvider N-read provider could reuse the batched form. This is real fragility, not a token-style duplicate a scanner would catch.
    - *Fix:* Extract a core firestore util: Stream<List<Event>> watchByIdStream({required Stream<List<String>> idStream, required CollectionReference ref, bool descending}) plus a single chunked<T>(List<T>, int) extension; delete the three private copies and have all fan-out sites delegate.
    - *Seen in:* lib/events/data/event_repository.dart, lib/events/data/saved_event_repository.dart, lib/clubs/data/clubs_repository.dart
- **`n_plus_one_and_per_tile_realtime_profile_reads`** (6 sites → rule `DATA-ACCESS-BATCH-001`): Sequential per-uid Firestore reads and per-list-tile live snapshot listeners for static display data
    - *Why:* fetchPublicProfiles awaits N per-uid gets sequentially (linear latency on event rosters), and several list/grid surfaces open one persistent realtime snapshot listener PER tile (or do N per-id fetchClub reads) purely to render static name/photo/date labels they never live-update. This wastes round-trips, opens N listeners that are never torn down per-item, and clubNameLookupProvider also leaks a non-autoDispose family instance per query. Batched/one-shot reads already exist (watchClubsByIds, fetchPublicProfiles, fetchEvent).
    - *Fix:* Parallelize fetchPublicProfiles with Future.wait over the deduped uid set; resolve roster/list profiles ONCE in a FutureProvider keyed by the id list and pass resolved values into tiles instead of per-tile watch streams; expose a one-shot fetchEvent provider for static labels; convert clubNameLookupProvider to autoDispose @riverpod backed by the batched whereIn read.
    - *Seen in:* lib/public_profile/data/public_profile_repository.dart, lib/clubs/presentation/club_name_lookup.dart, lib/reviews/presentation/reviews_history_screen.dart, lib/safety/presentation/settings_screen.dart, lib/swipes/presentation/event_recap_screen.dart
- **`uncached_image_network_no_decode_sizing_or_errorbuilder`** (16 sites → rule `IMAGE-NETWORK-PRIMITIVE-001`): Raw Image.network app-wide: no caching/decode-sizing, and some sites omit errorBuilder
    - *Why:* All ~16 network-image sites use raw Image.network with no cacheWidth/cacheHeight and no caching package, so avatars/thumbnails decode at full resolution and re-decode on scroll re-attach across rosters, chat, swipe cards. Separately, two sites omit the errorBuilder fallback their siblings provide, so a broken URL shows Flutter's default broken-image glyph. - **`run_to_event_rename_grammar_corruption`** (9 sites → rule `COPY-VERB-INTEGRITY-001`): Automated run->event rename corrupted user-facing copy ("You both event to make friends", "Long-event person")
    - *Why:* An automated rename replaced the verb 'run' with 'event' in user-visible compatibility-reason and emotional-run-tag labels, producing nonsense English rendered verbatim into Text widgets on the profile match-signals surface. ## functions/src (backend) — 89 verified findings (8 high / 21 med / 35 low)

> The Catch Cloud Functions backend (90 TS files, ~32k LOC) has a fundamentally sound authority model: callables consistently enforce App Check + requireAuth, host/owner authority is derived server-side (never from client claims), payment amounts are computed server-side, signature verification is timing-safe, and every domain collection is locked to `allow write: if false` so callables/webhooks are the sole write path. The high-severity findings are concentrated in three areas. (1) Payment reconciliation: there is NO Razorpay webhook, so a payment captured after the client gives up is never reconciled (charged-but-unbooked-and-unrefunded), and when an automatic refund fails the payment is silently recorded as status "completed" with no recovery path — the same anti-pattern exists on the Stripe path. (2) Trust-boundary gaps: updateUserProfile stores client-supplied profile-photo storagePath/url with no ownership validation, which the account-deletion sink then deletes via admin SDK (cross-user Storage deletion); createPublicClubReview accepts fully unauthenticated writes that poison trusted club rating aggregates; account deletion runs destructive Storage delete before re-runnable Firestore work with no compensation or tombstone re-entry guard. (3) Moderation bypass: blocked chat content is still delivered verbatim in the FCM push (moderation only redacts the stored doc via a separate racing trigger), and the banned text persists in lastMessagePreview. Cross-cutting themes dominate the rest: at-least-once triggers/webhooks lacking idempotency receipts; non-transactional read-modify-write on backend-owned denormalized counts; ~15 callables silently inheriting the 30/min default rate limit; unbounded collection scans / trigger write-amplification on hot analytics paths; client-supplied identifiers used as document ids/paths; unbounded free-text inputs; PII and raw upstream error bodies leaking to logs/clients; and canonical documentation (functions/README.md) that omits the entire live Stripe surface and ~30 other exports. Recommended priorities: ship a Razorpay webhook + a distinct non-recoverable refund state + reconciliation sweep; add ownership validation on storage paths and harden the deletion sink; gate the public review aggregate to verified/published; move moderation into the write boundary so the push cannot leak; introduce a shared event-receipt idempotency primitive for triggers/webhooks; and codify rules for rate-limit registration, atomic count mutation, and client-supplied id/path validation.
