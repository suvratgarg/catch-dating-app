# Widget Consolidation Pipeline — Execution Spec

## 0. Purpose and success criteria

`lib/**` contains **1,058 public widget classes** (see
`docs/audit_registry/widget_classification.json`). The no-private-widgets rule
worked: everything is public and ~all of it is catalogued in Widgetbook
(1,116 use-cases). The new problem is duplication and taxonomy drift:

- Suffix census (2026-07-02): 80 `*Card`, 24 `*Tile`, 17 `*Panel` (three names,
  likely one concept); 17 `*Pill`, 8 `*Chip`, 9 `*Badge`, 10 `*Label` (four
  names); 41 `*Section`, 39 `*Body`, 15 `*Content`.
- Known near-duplicate families visible from names alone:
  `ChatShareCard`/`ClubShareCard`/`EventShareCard`, `DarkPill` vs
  `EventSuccessDarkPill`, `CountdownBeatPill`/`CountdownCuePill`,
  `DashboardFocusLoadingCard`/`DashboardStrideLoadingCard`/`ClubDirectorySkeletonCard`.

This spec defines a **deterministic pipeline** that finds duplicate/near-
duplicate widgets mechanically and packages them into small, reviewable
**cluster packets**. A human + senior-model review pass then makes the actual
merge/rename/keep decisions. The executing agent for this spec does the
mechanical work only.

**Success is measured by drift surfaced and later fixed, not by a green
baseline.** Do not tune thresholds to make reports smaller. Report honest
numbers. (This is a standing owner principle for this repo.)

### Explicit non-goals for the executing agent

The executing agent must **NOT**:

1. Merge, rename, delete, or change the API of any widget.
2. Decide which widgets are duplicates "enough" to consolidate — the pipeline
   *proposes*, the review pass *decides*.
3. Ratify the naming lexicon (§8.2). It is included as a draft only.
4. Edit `packages/catch_ui_lints/**` (see §1 gotchas).
5. Change similarity thresholds from the defaults without running the
   calibration procedure (§4.4) and recording the calibration table.

Anything on this list that seems necessary → stop and escalate to the owner.

## 1. Environment facts and known gotchas (read before touching anything)

1. **Flutter SDK**: not on the default shell PATH. Use the repo wrappers
   `tool/flutter_with_env.sh` / `tool/firebase_with_env.sh`, or
   `export PATH="$HOME/development/flutter/bin:$PATH"` inline. Dart is at the
   same location (`flutter` bundles `dart`).
2. **Never edit `packages/catch_ui_lints/`**: recompiling the analyzer plugin
   crashes `dart analyze` locally. Running *standalone* Dart scripts that
   depend on `package:analyzer` is fine — only the analysis-server plugin
   recompile is broken. All new enforcement in this spec is grep/script-based,
   not lint-plugin-based.
3. **`rg` has no lookahead** — `(?!...)` patterns silently match nothing.
   Design-token names contain digits (`CatchSpacing.s5`, `micro10`), so naive
   `[0-9]` patterns false-positive. `//` comments match `[-+*/]` patterns.
   Anchor literals, exclude comments, and **spot-check every count you report**
   (past audits found grep counts that were 100% false positives).
4. **`flutter analyze` has no `--format` flag.** Parse text output if needed.
5. **Repo tool conventions** (follow them exactly):
   - Node scripts live in `tool/<area>/*.mjs`, import
     `fromRepo`/`relativeToRepo` from `tool/lib/repo_paths.mjs`, support
     `--check` / `--json` / `--write <path>` flags, and print usage on
     `--help`.
   - Every new tool is registered in `tool/tools_manifest.json` (see existing
     entries for the shape: `id`, `category`, `path`, `command`, `safety`,
     `status`, `description`, `checks`).
   - Gates ship with **seeded probes** (anti-vacuity): fixtures containing a
     planted violation that the gate MUST flag, run before the real gate. Gold
     standard: `tool/check_catch_ui_lints.sh`.
6. **Widgetbook workspace** is `widgetbook/` (own pubspec, depends on the app
   via path). Use-cases are `@widgetbook.UseCase(name:, type:, path:)`
   annotated top-level functions; `widgetbook/lib/main.directories.g.dart` is
   generated by `widgetbook_generator` via `build_runner`.
7. **Docs hygiene**: date-stamped outputs are snapshots; registries under
   `docs/audit_registry/` carry `version`/`updated` headers. Register this doc
   and its artifacts per `docs/README.md` conventions when done (do not create
   duplicate trackers).

## 3. Pipeline overview

```
Phase A  extract_fingerprints (Dart, AST)   → artifacts/widget_dedupe/fingerprints.json   (local)
Phase B  cluster + rank (.mjs)              → docs/audit_registry/widget_similarity.json  (checked in)
Phase C  render use-cases + perceptual hash → artifacts/widget_dedupe/renders/*.png (local)
                                            → visual edges merged into widget_similarity.json
Phase D  cluster review packets (.mjs)      → docs/design_parity/widget_consolidation/clusters/*.md
Phase E  decision ledger (schema only)      → docs/design_parity/widget_consolidation/decisions.json
Phase F  ratchets: similarity gate, naming census, build-helper burn-down
```

Two independent detectors, because they catch different failure modes:

- **Structural** (Phase A/B): "coded the same" — rename-invariant AST
  fingerprints. Catches copy-paste-and-tweak.
- **Visual** (Phase C): "renders the same" — perceptual hashes of Widgetbook
  use-case renders. Catches independently-written lookalikes that ASTs miss.

Phases A, B, D are required. Phase C is required-with-fallback (§5.4). Phase E
is schema + validation only. Phase F ships report-only first, then the owner
decides when to make gates blocking.

## 4. Phase A — Structural fingerprint extraction (Dart)

### 4.1 Tool location and setup

New standalone Dart mini-package: `tool/widget_dedupe/`

```
tool/widget_dedupe/
  pubspec.yaml          # name: widget_dedupe; deps: analyzer ^9.0.0, path, crypto, args
  bin/extract_fingerprints.dart
  lib/src/...           # extraction logic, unit-testable
  test/                 # unit tests incl. seeded probe fixtures
  fixtures/             # probe dart files (§9)
```

`analyzer ^9.0.0` matches `packages/catch_ui_lints`. This package is run with
`dart run` only — it is **not** an analyzer plugin and must not be added to any
`analysis_options.yaml`.

CLI:

```
dart run tool/widget_dedupe/bin/extract_fingerprints.dart \
  [--files <comma-separated dart files>]   # default: all widgets in classification registry
  [--out <path>]                           # default: artifacts/widget_dedupe/fingerprints.json
```

### 4.2 Input set

Read `docs/audit_registry/widget_classification.json`. Fingerprint every entry
with `classKind: widget` (all roles, including `screen` — screens are clustered
separately in Phase B, not dropped).

**State-class linking**: for `StatefulWidget`/`ConsumerStatefulWidget`, the
real `build()` lives in the (private) State class. Link State → widget via the
`createState()` return type; fall back to the `_FooState` → `Foo` name
convention. The State class's widget-returning methods are folded into the
owning widget's fingerprint. `widget-state` entries get no fingerprint of
their own.

Parse with the **parsed (unresolved) AST** (`parseFile` /
`AnalysisContextCollection` without resolution). No type resolution is needed —
everything below is syntactic. This keeps the tool fast and immune to the
plugin-recompile problem.

### 4.3 Normalization → token stream

For each widget class, collect methods in this deterministic order: `build`
first, then every other method whose return type is syntactically `Widget`,
`Widget?`, `List<Widget>`, or `PreferredSizeWidget`, sorted by name. (These
helper methods are themselves rule violations — record their presence in a
`hasWidgetHelpers` flag — but they still carry structure that must count
toward the fingerprint.)

Walk each method body pre-order and emit tokens:

| AST node | Emitted token | Notes |
|---|---|---|
| `InstanceCreationExpression`, or `MethodInvocation` whose method name starts with an uppercase letter | `W:<Name>` | The widget-construction skeleton. Use the constructor name incl. named constructors (`W:CatchButton.ghost`). |
| Each argument label of the above | fine stream: `A:<label>` in source order; **coarse stream: labels sorted within each `ArgumentList`**; positional args emit `A:_` | v0.2.0: source-order labels made argument permutations look like different code (measured: killed the `DarkPill` family match). Sort per `ArgumentList` node in the coarse stream. |
| `PrefixedIdentifier` / `PropertyAccess` whose prefix is a design-token class | fine stream: `T:<Class>.<member>` verbatim; **coarse stream: `T:<Class>` only** | Token classes: everything under `lib/core/theme/` — enumerate at runtime (do not hardcode). v0.2.0: verbatim members are feature-namespaced (`CatchOpacity.manualQaPillFill` vs `.revealSurfaceFill`), so cross-feature clones could never match; measured impact on the `DarkPill` pair: Jaccard 0.087 → 0.44 after coarsening. Verbatim members still power the packet "token deltas" section. |
| String literal | `S` | Rename-invariant. |
| Numeric literal | `N` | |
| Boolean literal | `B` | |
| `ConditionalExpression` | `C:cond` | |
| `IfStatement` / collection-`if` | `C:if` | |
| loops / collection-`for` | `C:for` | |
| Spread element | `C:spread` | |
| `null` literal | `Z` | |
| Everything else (local identifiers, params, method calls starting lowercase) | dropped | This is what makes fingerprints rename-invariant. |

Notes:

- Comments and whitespace vanish for free in a parsed AST.
- Do **not** attempt to inline calls to other widgets or private helpers in
  other classes; a `W:` token for the callee is the correct granularity.
- **v0.2.0: emit BOTH streams per widget** — the *fine* stream (verbatim `T:`
  members, source-order `A:` labels) for `shapeHash` and packet diffs, and the
  *coarse* stream (class-only `T:`, sorted `A:` runs) for all similarity math.

### 4.4 Fingerprint representations

Per widget, compute and record:

1. **`shapeHash`** — sha256 of the **fine** token stream joined with `\n`.
   Identical hash ⇒ structural clone (only literals/identifiers differ).
   Also record **`coarseShapeHash`** over the coarse stream.
2. **`shingles`** — the set of **2-gram** shingles over the **coarse** token
   stream (v0.2.0: was 3-grams over the fine stream; 3-grams amplified every
   local difference threefold and, combined with fine tokens, drove
   ground-truth family Jaccard down to 0.09–0.15), each stored as the first 10
   hex chars of its sha1. Stored in the **local** artifact only.
3. **`tokenMultiset`** — the coarse stream as a token→count map (k=1). Used by
   the small-widget detector (§5.2): shingle Jaccard is meaningless for
   streams of 10–30 tokens, which is where compact pills/badges/labels live.
4. **`simhash128`** — 128-bit SimHash over the coarse shingle set (feature =
   shingle sha1 digest bits, weight 1). Stored in the **checked-in** registry;
   used by the CI ratchet (§8.1) so the registry stays small.
4. Metadata: `name`, `file`, `role` (copied from classification), `loc`
   (class declaration line span), `constructorParams`
   (`[{name, type, required}]` for the default constructor),
   `widgetsUsed` (distinct `W:` names), `tokensUsed` (distinct `T:` values),
   `tokenStreamLength`, `hasWidgetHelpers`, `stateClass` (nullable).

**Calibration procedure (required before Phase B thresholds are trusted):**
compute exact Jaccard for a random sample of 2,000 widget pairs plus all pairs
sharing a name suffix; plot/aggregate SimHash hamming distance vs Jaccard into
a small table (`hamming bucket → mean/min/max Jaccard`). Record the table in
the `widget_similarity.json` header. v0.2.0: calibrate the hamming threshold
so it over-approximates (never misses) coarse Jaccard ≥ 0.55; the v0.1.0
calibration (hamming 18 against fine 3-gram shingles) is void because the
shingle basis changed. Do not adjust the Jaccard thresholds themselves.

### 4.5 Output

`artifacts/widget_dedupe/fingerprints.json` (add `artifacts/widget_dedupe/` to
`.gitignore`):

```jsonc
{
  "version": 1,
  "generatedAt": "2026-07-02T…",
  "tokenClassSource": "lib/core/theme",
  "widgets": [ { /* fields from §4.4, incl. shingles */ } ],
  "failures": [ {"name": "...", "file": "...", "reason": "..."} ]
}
```

### 4.6 Acceptance criteria — Phase A

- Fingerprints produced for **≥ 95%** of `classKind: widget` entries;
  every failure listed with a reason (no silent drops).
- Deterministic: two consecutive runs produce byte-identical output modulo
  `generatedAt`.
- Unit tests: normalization goldens for at least 5 real widgets (checked-in
  expected token streams), State-linking test, helper-method-folding test.
- Seeded probes (§9) pass.

## 5. Phase B — Clustering and the similarity registry (.mjs)

### 5.1 Tool

`tool/design/build_widget_similarity.mjs` (usage/`--check`/`--json` per repo
conventions). Reads `artifacts/widget_dedupe/fingerprints.json`, writes
`docs/audit_registry/widget_similarity.json` (checked in).

### 5.2 Detectors, edges, and clusters (v0.2.0 — supersedes the 0.1.0 threshold scheme)

The v0.1.0 run proved that a single thresholded structural detector has a
precision–recall wall: honest thresholds found only 7 clusters, and 2 of the
4 ground-truth families are *conceptual* duplicates (same design idea,
genuinely different subtree structure) that no structural threshold reaches
before random-pair noise floods in (measured: ground-truth pairs at coarse
Jaccard 0.20–0.48 vs random-pair p99.9 = 0.40 over 5,000 samples). The review
queue is therefore built from **four detectors, ranked, not thresholded**;
thresholds exist only where auto-clustering or CI gating demands precision.

All similarity math runs on the **coarse** stream (§4.3/4.4). Pair count is
~1058² / 2 ≈ 560k exact computations — brute force, no approximations.

**Detector 1 — structural clusters (auto-cluster, high precision):**
- Exact edges: identical `shapeHash` or `coarseShapeHash`.
- Strong edges: coarse 2-gram Jaccard ≥ **0.55** (measured random-pair rate
  ~0.02%).
- Small-widget edges: both coarse streams < 40 tokens and `tokenMultiset`
  Jaccard ≥ **0.75** (measured: catches `DarkPill`/`EventSuccessDarkPill` at
  0.81; random small-pair rate 0.1%).
- Clusters = connected components over these edges, `screen` roles clustered
  separately; cross-role edges recorded with flag `cross-role`.
- Widgets with coarse stream < 8 tokens are skipped (`skippedTrivial`).

**Detector 2 — ranked structural pairs (no threshold):** the global **top 200
pairs** by `max(jaccard, containment)` where containment =
`|A∩B| / min(|A|,|B|)` on coarse shingles. Containment is directional — it
catches "small widget rebuilt inside a bigger one" and "feature widget vs core
primitive" (measured: `ChatShareCardSheet` vs the core `CatchShareCardSheet`
scores containment 0.69 — invisible to symmetric Jaccard at 0.49). These pairs
are **advisory**: they feed packets, never auto-merge clusters.

**Detector 3 — name families:** widgets sharing their final **two** CamelCase
segments (e.g. `ShareCard`, `MetaRow`, `LoadingCard`), after stripping
known feature prefixes, with ≥ 2 members. One-segment families (all `*Card`)
are too coarse — do not use. This detector is what actually finds conceptual
duplicates: all four ground-truth families are name-discoverable and only one
is structurally discoverable. High precision, zero cost.

**Detector 4 — visual edges** (Phase C, §5.4): unchanged.

Every packet (§7) records which detectors fired for it. A family flagged by
two or more detectors outranks single-detector findings.

### 5.3 Enrichment per cluster

- **`usageCount`** per member: `rg -w '<WidgetName>'` across `lib/` and
  `widgetbook/lib/` (excluding `*.g.dart` and the defining file), raw match
  count. Record as `usageCountRaw` — it is a grep count, so also record
  `usageCountNote: "raw rg -w count; may include name collisions"`.
  **Spot-check the 20 highest counts manually and record the receipt** (§10).
- **`absorbCandidate`**: true if any member lives under `lib/core/widgets/` or
  has a `contractId` in `catch.components.json`. These clusters mean "a
  feature rebuilt an existing primitive" — rank boost ×2.
- **`paramCompatibility`**: pairwise Jaccard over constructor param *names*;
  report min/mean. (High structural + high param similarity = trivial merge;
  high structural + low param = needs API design taste.)
- **`cohesion`**: mean pairwise Jaccard among members.
- **`rank`**: `cohesion × (Σ usageCountRaw + memberCount) × (absorbCandidate ? 2 : 1)`,
  ties broken by cluster slug. Deterministic.
- **`nameSignal`**: shared name stem after stripping feature prefixes
  (longest common suffix of CamelCase segments, min 1 segment, e.g.
  `ShareCard`), or null.

### 5.4 Merging visual edges (after Phase C)

Phase C adds edges `{a, b, signal: "visual", hamming}` for use-case renders
whose 64-bit dHash hamming distance ≤ 6. A visual edge between widgets in
different structural clusters merges nothing automatically — it adds a
`visualBridge` entry to both clusters and a rank boost of +25% each. (Visual
evidence is coarse — use-case pages often contain many state cards — so it
raises priority, never merges clusters on its own.)

If Phase C is skipped/fails (§6.5), the registry records
`"visualSignal": "unavailable: <reason>"` and everything else proceeds.

### 5.5 Registry schema

```jsonc
{
  "version": 1,
  "updated": "YYYY-MM-DD",
  "params": {"jaccardStrong": 0.55, "smallWidgetMultisetJaccard": 0.75,
              "smallWidgetMaxTokens": 40, "rankedPairCount": 200, "shingleK": 2,
              "stream": "coarse", "simhashHammingStrong": <recalibrated>,
              "visualHammingStrong": 6},
  "calibration": [ {"hammingBucket": "0-4", "pairs": n, "jaccardMin": …, "jaccardMean": …}, … ],
  "summary": {"widgets": n, "exactClusters": n, "clusters": n, "relatedEdges": n,
               "screensClusters": n, "skippedTrivial": n, "absorbCandidates": n},
  "widgets": [ {"name", "file", "role", "shapeHash", "simhash128",
                 "tokenStreamLength", "usageCountRaw", "hasWidgetHelpers"} ],
  "clusters": [ {"id": "c001-share-card", "rank": 1, "score": …, "cohesion": …,
                  "absorbCandidate": true, "nameSignal": "ShareCard",
                  "detectors": ["structural", "name"],
                  "members": ["ChatShareCard", "ClubShareCard", "EventShareCard"],
                  "paramCompatibility": {"min": …, "mean": …},
                  "visualBridges": [], "scope": "widget" | "screen"} ],
  "rankedPairs": [ {"a", "b", "jaccard", "containment", "flags": []} ],
  "nameFamilies": [ {"stem": "ShareCard", "members": [...], "usageSum": n} ]
}
```

### 5.6 Acceptance criteria — Phase B

- Registry checked in; `--check` mode verifies the checked-in file matches a
  regeneration (modulo `updated`), same pattern as
  `generate_widget_variant_inventory.mjs --check`.
- **Ground-truth recall table (v0.2.0)**: each of the four §0 families PLUS
  the absorb pair `ChatShareCardSheet`/`CatchShareCardSheet` must surface
  through at least one detector, and the registry must record which:

  | ground truth | expected detector(s) |
  |---|---|
  | `DarkPill` / `EventSuccessDarkPill` | small-widget structural + name |
  | `ChatShareCardSheet` / `CatchShareCardSheet` | ranked pairs (containment) |
  | `*ShareCard` (3 members) | name family |
  | `CountdownBeatPill` / `CountdownCuePill` | name family (structurally distinct — verified) |
  | `Dashboard*LoadingCard` / `ClubDirectorySkeletonCard` | name family + ranked pairs |

  A miss in the *expected* column is a bug; a hit through an additional
  detector is fine. Do not force edges to satisfy this table.
- Summary numbers reported in the final handoff note **with** the spot-check
  receipts, not just totals.

## 6. Phase C — Visual fingerprints from Widgetbook renders

### 6.1 Render harness

New test in the widgetbook workspace:
`widgetbook/test/render_use_cases_test.dart` (runs with
`flutter test` inside `widgetbook/`; it is a capture harness, not an
assertion suite — follow the `test/ui_captures/` pattern from the app,
including the `pixelRatio`-into-`toImage` fix).

- Import `main.directories.g.dart`; walk the `directories` tree
  (`WidgetbookFolder` / `WidgetbookComponent` / `WidgetbookUseCase` nodes).
- For each use-case: `pumpWidget` wrapping the use-case builder in
  `ProviderScope` + `MaterialApp(theme: AppTheme.light)` at surface size
  390×844, `pixelRatio: 1.0` (hash stability beats fidelity), settle with a
  bounded `pump` loop (do not `pumpAndSettle` unbounded — some use-cases have
  infinite animations; cap at ~10 pumps of 100ms).
- Capture via `RepaintBoundary`/`toImage`, write PNG to
  `artifacts/widget_dedupe/renders/<sanitized-path>__<usecase>.png`.
- Any exception → record in `artifacts/widget_dedupe/render_failures.json`
  (`{path, useCase, error}`) and continue. Never let one use-case kill the
  run.
- Env toggle `WIDGET_DEDUPE_FILTER=<substring>` to render a subset while
  iterating.

### 6.2 Use-case → widget mapping

Parse `@widgetbook.UseCase(… type: <WidgetName> …)` annotations from
`widgetbook/lib/**` source (regex on `type:\s*(\w+)` within the annotation is
acceptable — the annotation form is uniform). Emit
`artifacts/widget_dedupe/usecase_widget_map.json`. Use-cases whose `type` is a
catalog scaffold rather than a lib widget map to null and are excluded from
visual edges.

### 6.3 Perceptual hashing

`tool/design/compute_visual_hashes.mjs`:

- Decode PNG with `pngjs` (pure JS — do not add native deps like `sharp`).
- Grayscale → resize to 9×8 (nearest-neighbor box sampling is fine) → 64-bit
  **dHash** (horizontal gradient).
- Output `docs/audit_registry/widget_visual_hashes.json` (checked in — it is
  small: one 16-hex-char hash + ids per use-case, plus a render-failure count).
- Emit edges (hamming ≤ 6) into Phase B via
  `build_widget_similarity.mjs --visual docs/audit_registry/widget_visual_hashes.json`.

### 6.4 Acceptance criteria — Phase C

- ≥ 70% of use-cases render successfully; every failure is in the failure
  ledger with an error string. If the rate is below 70%, diagnose the top 3
  failure categories before deciding (with the owner) whether to fix or
  accept.
- Two renders of the same use-case produce the same dHash (determinism check
  on a 20-use-case sample; animations capped as above).
- Probe pair (§9) gets a visual edge.

### 6.5 Fallback

If pumping generated directories proves structurally impossible (e.g. builders
require live Firebase), do not sink more than ~1 day into it: ship Phases A/B/D
without visual signal, record the blocker in the registry header and in the
handoff note, and move on. Structural signal alone is sufficient for the
first review round.

## 7. Phase D — Cluster review packets (the human/senior-model handoff)

### 7.1 Generator

`tool/design/build_consolidation_packets.mjs` — reads
`widget_similarity.json` + fingerprints + renders, writes:

```
docs/design_parity/widget_consolidation/
  INDEX.md                      # ranked checklist table
  clusters/c001-share-card.md
  clusters/c002-….md
```

Scope v1 (v0.2.0): packets are generated from all four detector outputs,
deduplicated (a widget set already covered by a higher-ranked packet is not
re-emitted):

1. every structural cluster (`scope: widget`; screens are round 2);
2. every `absorbCandidate` cluster and every ranked pair whose containment
   ≥ 0.75 where one side is a `lib/core/widgets/` or contracted primitive;
3. every name family with ≥ 2 members, merged with its members' structural
   data;
4. the remaining top ranked pairs, until the total packet count reaches ~80.

Packets are ordered by: detector count desc, then rank score. Each packet
header lists its detectors (e.g. `detectors: name + structural`).

### 7.2 Packet format (this is the contract for the review pass — follow exactly)

Each packet must be reviewable **without opening any other file**:

````markdown
# c001 — ShareCard family                          rank 1 · score 214.2

| member | file | role | usage | LOC | params | tokens | widgetbook |
|---|---|---|---|---|---|---|---|
| ChatShareCard | lib/matches/… | composition | 11 | 84 | 6 | 9 | yes |
| …

**Similarity**: cohesion 0.87 · exact-shape subset: {ClubShareCard, EventShareCard}
**Param compatibility**: min 0.55 mean 0.71 — union of params: `title, subtitle, imageUrl, onShare, tone*, footer*` (`*` = not in all members)
**Token deltas**: ChatShareCard uses `T:CatchSpacing.s3` where others use `T:CatchSpacing.s4`
**Absorb candidate**: CatchShareCardSheet (lib/core/widgets/catch_share_card_sheet.dart, contract catch.share_card_sheet)
**Visual**: renders at artifacts/widget_dedupe/renders/… (local-only, regenerate with `flutter test widgetbook/test/render_use_cases_test.dart`)

## ChatShareCard — constructor
````

## 8. Phase E + F — Decision ledger, naming census, ratchets

### 8.1 Decision ledger and similarity ratchet

`docs/design_parity/widget_consolidation/decisions.json`:

```jsonc
{
  "version": 1,
  "decisions": [
    {"clusterId": "c001-share-card",
     "decision": "absorb-into-existing",       // merge-new-primitive | absorb-into-existing | keep-distinct | rename-only | delete
     "canonicalSymbol": "CatchShareCardSheet",
     "members": [{"name": "ChatShareCard", "action": "absorb"},
                  {"name": "ClubShareCard", "action": "absorb"}],
     "rationale": "…", "decidedBy": "owner+claude", "date": "YYYY-MM-DD"}
  ]
}
```

The executing agent creates the empty ledger + a validator
(`tool/design/check_widget_consolidation.mjs --check`) that asserts: every
ledger entry references a real cluster and real members; every packet whose
Decision section is filled has a matching ledger entry. It does **not**
require decisions to exist (that gate comes later, owner's call).

**Similarity ratchet** (the duplication-regression gate),
`tool/design/check_widget_similarity.mjs --check`:

1. Determine widget classes added/renamed in the working tree vs the registry
   (by name+file against `widget_similarity.json.widgets`).
2. For each new widget, run Phase A extraction on just its file
   (`--files`), compare `simhash128` against all registry entries.
3. **Fail** if hamming ≤ calibrated-strong threshold against any existing
   widget, with the message:
   `NewWidget is structurally ~X% similar to ExistingWidget (lib/…). Reuse it, extend it, or record a keep-distinct decision in decisions.json.`
4. The escape hatch is a `keep-distinct` ledger entry naming both widgets —
   **not** a source comment. (Governance through the ledger, consistent with
   the owner's drive to reduce scattered `allow:` markers.)
5. Ships **report-only** (exit 0, prints findings) until the owner flips a
   `--enforce` flag in CI. Register in `tool/tools_manifest.json`; propose —
   do not commit — the `.github/workflows/flutter-ci.yml` wiring in the
   handoff note (that file is contended on current branches).

### 8.2 Naming census + draft lexicon (census is mechanical; ratification is not)

`tool/design/generate_widget_name_taxonomy.mjs` →
`docs/audit_registry/widget_name_taxonomy.json`: for every public widget,
split CamelCase, record `{finalNoun, qualifiers, featurePrefix, role}`;
aggregate counts per final noun and per (noun, role).

**Draft lexicon — DO NOT ENFORCE; to be ratified in the review pass:**

| Canonical noun | Meaning | Absorbs / bans |
|---|---|---|
| `Card` | Bordered/elevated content container, tappable or not | `Panel`, `Tile` (when container-like) |
| `Tile` | Fixed-height list row inside a scrolling list | `Row` used as list item |
| `Chip` | Compact **interactive/selectable** element | `Pill` (interactive uses) |
| `Badge` | Compact **non-interactive** status/count marker | `Pill` (static uses), `Label` (status uses) |
| `Section` | Titled vertical grouping within a page | `Body`, `Content` (when they mean "grouped chunk") |
| `Header` / `Footer` | Leading/trailing furniture of a section, sheet, or screen | — |
| `Sheet` | Bottom-sheet surface | — |
| `Banner` | Full-width inline notice | — |
| `Screen` | Routed page | `Page`, `View` |
| `State` (suffix `…State` on widgets like `CatchErrorState`) | Full-bleed empty/error/loading placeholder | `Placeholder` |

Naming grammar (draft): `Catch<Noun><Qualifier?>` for design-system primitives
(must live in `lib/core/widgets/` and have a contract);
`<Feature><Domain?><Noun>` for feature composites (must be composed from Catch
primitives). Enforcement, once ratified, is a suffix-whitelist `.mjs` gate
shipped report-only first.

## 9. Seeded probes (anti-vacuity — required for every gate)

Fixtures in `tool/widget_dedupe/fixtures/` (excluded from app analysis; not
under `lib/`):

- `probe_dupe_a.dart` / `probe_dupe_b.dart`: two widget classes with different
  class/param/variable names but identical structure (copy a real ~40-line
  card, rename everything). MUST produce identical `shapeHash`.
- `probe_near_c.dart`: same as A with one extra child widget and one changed
  token. MUST land in Jaccard band [0.65, 1.0) relative to A — assert a strong
  or related edge exists.
- `probe_distinct.dart`: a structurally unrelated widget. MUST NOT edge to A/B/C
  above the related threshold.
- Visual probe: two Widgetbook use-cases in the harness's test fixtures
  rendering identical output under different names → dHash hamming 0.

Harness: `tool/design/check_widget_dedupe_probes.mjs` runs extraction +
clustering over the fixtures and asserts all four properties; wire it as a
`checks` entry for every tool registered in `tool/tools_manifest.json`, and
run it at the start of `check_widget_similarity.mjs --check` (a gate that
can't catch its own probe must fail loudly, per the
`tool/check_catch_ui_lints.sh` gold standard).
