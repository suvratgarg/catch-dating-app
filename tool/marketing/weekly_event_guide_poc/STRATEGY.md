# Weekly Event Guide Strategy

## Recommendation

Start with a city-first weekly guide for Mumbai, then organize the slides by
activity mix inside that city. Do not start with a global activity-first account
or a website guide. The simplest repeatable product is:

```text
Mumbai events that are singles-friendly this week
```

Use `singles-friendly` as the public default while events are mostly non-Catch
inventory. It is broader, easier to defend, and less likely to misrepresent a
third-party event. Use `singles social events` only when the event itself is
explicitly singles-led, dating-led, mixer-led, or hosted by Catch.

## Format

Primary format: Instagram carousel.

Supporting formats can be generated from the same packet later:

- Instagram story frames
- Short newsletter block
- Website guide
- In-app discovery module

The first operating loop should prove the carousel because it has the strongest
fit with visual discovery, saves/shares, host visibility, and low production
friction.

## Grouping

Start city-first:

```text
Mumbai / this week / 5-7 recommended events
```

Inside each city guide, maintain a mixed activity slate:

- culture
- food and drink
- active or outdoor
- learning or workshops
- nightlife or music

This prevents the content from becoming "restaurant recommendations with a
dating caption." The editorial promise is social discovery, not generic city
content.

## CTA

Use a dual CTA until Catch has enough owned event supply:

```text
Join the Catch waitlist.
Hosting something social? Submit it for review.
```

The host CTA should be explicit about review, not automatic featuring. The
promise is visibility if the event is a good fit, not guaranteed promotion.

Avoid making the website guide the primary CTA for now. It adds surface area
before the supply and review loop is proven.

## AI Boundary

AI should act as a copilot, not an autopublisher.

Allowed:

- expand source/search config into a weekly research plan
- normalize event facts into a common schema
- score candidates with visible reasons
- draft carousel copy in multiple tones
- identify missing fields and verification gaps
- propose image briefs for a designer or renderer

Not allowed:

- publish without human approval
- imply Catch hosts a third-party event
- invent event facts
- use unofficial Instagram scraping as a core source
- present sample candidates as verified public recommendations

## Source Strategy

Use three source classes:

1. Catch-owned submissions: highest trust, future priority.
2. Open web sources and event platforms: useful if their terms and robots
   policies permit the intended use.
3. Instagram and publisher posts: manual-reference leads unless using an
   approved official API path.

Every public event should carry source evidence for:

- event name
- date and time
- venue or neighborhood
- price or RSVP requirement
- host or organizer
- source URL
- why it is singles-friendly or singles-led

## Effort Model

V0 manual pilot:

- 2-4 hours per city per week.
- Best for testing tone, format, saves/shares, and host interest.
- No source adapters required.

V1 copilot:

- 30-90 minutes per city per week after setup.
- Source/search config produces a candidate queue.
- Human verifies, edits, approves, and exports.

V2 operating system:

- Source snapshots, citations, editor UI, image renderer, and publish handoff.
- Still human-approved.
- Can be cloned for other funnels once the Mumbai loop is stable.

## Success Signals

Audience signals:

- saves per reach
- shares per reach
- profile visits
- waitlist clicks
- replies asking for more events

Host signals:

- host submission clicks
- inbound DMs from organizers
- repeat host submissions
- featured hosts reposting the carousel

Editorial signals:

- percentage of candidates approved
- time from source scan to final packet
- number of corrections after draft
- source reliability by source ID
